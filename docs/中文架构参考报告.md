# Healthcare AI 项目架构参考报告

## 1. 项目目标

本项目面向 healthcare 场景，当前目标是实现一个医生可用的症状查询辅助系统。医生输入症状或病情描述后，系统通过多智能体流程完成输入结构化、医学知识检索、大模型辅助分析和报告汇总。

当前系统定位不是自动诊断系统，而是 clinical decision support，即临床决策辅助系统。输出结果应作为医生参考，不能替代医生诊断。

## 2. 当前代码架构

当前 Python 侧架构如下：

```text
医生输入症状
  -> app.main / FastAPI
    -> Orchestrator
      -> AgentCoordinator
        -> SymptomExtractionAgent
        -> MedicalKnowledgeAgent
          -> RagMcpClient
          -> mcp_servers/rag_server.py
          -> 医学知识库
        -> DifferentialDiagnosisAgent
          -> DeepSeek / OpenAI-compatible LLM
        -> EvidenceReviewAgent
          -> DeepSeek / OpenAI-compatible LLM
        -> SafetyCheckAgent
        -> ReportAgent
          -> DeepSeek / OpenAI-compatible LLM
```

各模块职责如下：

| 模块 | 代码位置 | 功能 |
|---|---|---|
| Orchestrator | `app/orchestrator.py` | 系统总入口，接收一次症状查询任务 |
| AgentCoordinator | `app/agents/coordinator.py` | 按顺序调度多个 agent |
| SymptomExtractionAgent | `app/agents/symptom_extraction_agent.py` | 将医生输入的症状文本做基础结构化 |
| MedicalKnowledgeAgent | `app/agents/medical_knowledge_agent.py` | 调用 RAG/MCP 检索医学知识库 |
| DifferentialDiagnosisAgent | `app/agents/differential_diagnosis_agent.py` | 调用大模型生成候选鉴别诊断 |
| EvidenceReviewAgent | `app/agents/evidence_review_agent.py` | 调用大模型结合 RAG 证据评估候选疾病 |
| SafetyCheckAgent | `app/agents/safety_check_agent.py` | 本地规则检查红旗征和急诊风险线索 |
| ReportAgent | `app/agents/report_agent.py` | 调用大模型汇总前面 agent 的输出，生成最终医生报告 |
| RagMcpClient | `app/mcp_clients/rag_client.py` | 作为 agent 和 MCP server/知识库之间的接口 |
| rag_server | `mcp_servers/rag_server.py` | 预留医学知识检索工具 |

## 4. 为什么采用 Multi-Agent 架构

医疗场景下，医生的一次问题通常不是单一步骤可以完成的。例如：

```text
发热三天，咳嗽，胸闷，这些症状需要考虑哪些疾病？
```

这个问题至少包含以下子任务：

1. 提取症状和病情描述。
2. 检索医学知识或临床指南。
3. 基于症状和知识进行鉴别分析。
4. 给出红旗征、追问问题、建议检查。
5. 汇总为医生可读的报告。

如果全部交给一个 LLM 一次性完成，系统难以控制，也难以定位错误来源。因此本项目将任务拆成多个 agent：

```text
SymptomExtractionAgent 负责输入结构化
MedicalKnowledgeAgent 负责知识检索
DifferentialDiagnosisAgent 负责候选鉴别诊断
EvidenceReviewAgent 负责结合 RAG 证据评估候选疾病
SafetyCheckAgent 负责红旗征检查
ReportAgent 负责最终报告汇总
```

这种设计与 MDAgents、MedAgents、KG4Diagnosis 这类医疗 multi-agent 工作的思路一致：将复杂医疗任务拆分为多个角色或阶段，再由协调器统一组织。

## 5. 为什么使用 RAG

医疗领域不适合直接依赖大模型内部知识。原因包括：

1. 大模型可能产生 hallucination。
2. 医学知识会更新。
3. 医疗输出需要可追溯依据。
4. 不同医院或科室可能有自己的指南和规范。

因此系统设计为：

```text
医生问题
  -> 检索医学知识库
  -> 将检索结果交给 LLM
  -> LLM 基于检索内容生成辅助分析
```

这个思路与 Almanac、npj Digital Medicine 中的 guideline RAG framework、SurgeryLLM 等工作一致。

当前代码已经预留：

```text
MEDICAL_KNOWLEDGE_BASE=E:\path\to\your\approved_medical_docs.json
```

后续可以将该 JSON 知识库替换为：

1. 向量数据库。
2. 医院内部指南。
3. 临床路径文档。
4. 药品说明书。
5. 专科知识库。

## 6. MCP 在本项目中的作用

MCP 不是医学推理核心，而是工具和数据源的边界层。

当前设计为：

```text
Agent
  -> MCP Client
    -> MCP Server
      -> 数据源或外部服务
```

在本项目中，MCP 可以用于封装：

1. 医学知识库检索。
2. 患者信息查询。
3. 预约系统查询。
4. 检验检查结果查询。
5. Spring Boot 后端提供的业务接口。

这样做的好处是 agent 不需要直接关心数据库、HTTP 接口或具体系统实现，只需要调用统一工具。

## 8. 后续扩展方案

第一阶段，当前已完成：

```text
症状输入
  -> multi-agent 调度
  -> DeepSeek API 多阶段调用
  -> 缺知识库时明确提示
```

第二阶段，建议接入真实医学知识库：

```text
MEDICAL_KNOWLEDGE_BASE
  -> 医学指南 JSON
  -> MedicalKnowledgeAgent
  -> DifferentialDiagnosisAgent
  -> EvidenceReviewAgent
  -> ReportAgent
```

第三阶段，引入 Spring Boot：

```text
Spring Boot
  -> 账号管理
  -> 医生/患者管理
  -> 查询任务管理
  -> Kafka 发送 AI 任务
```

第四阶段，Python AI worker 通过 Kafka 处理任务：

```text
Spring Boot
  -> Kafka
    -> Python AI Worker
      -> Multi-Agent + RAG + LLM
    -> Kafka
  -> Spring Boot 保存结果
```

第五阶段，将 Spring Boot 服务包装为 MCP tools：

```text
get_patient_profile
get_lab_results
search_medical_knowledge
get_appointment_slots
```

## 9. 可用于汇报的总结

本项目采用 retrieval-augmented multi-agent clinical decision-support 架构。医生输入症状后，系统先由症状抽取 agent 对输入进行结构化，再由医学知识 agent 检索外部医学知识库，随后由大模型推理 agent 基于症状和检索证据生成辅助分析，最后由报告 agent 汇总输出。该架构参考了 Almanac、SurgeryLLM 等 RAG 医疗决策辅助系统，也参考了 MDAgents、MedAgents、KG4Diagnosis、MedLA 等医疗 multi-agent 推理框架。MCP 在系统中作为工具和数据源抽象层，方便后续接入 Spring Boot 后端、数据库、医学知识库和预约系统。
